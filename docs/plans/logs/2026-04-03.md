### real_mode 调试重大发现：CST COM Session 架构约束

#### 根本原因
`advanced_mcp`（modeler）和 `cst_results_mcp`（results）使用不同的 CST COM 对象：
- modeler：`cst.interface.DesignEnvironment.open_project()` - 非 interactive，创建独立会话
- results：`cst.results.ProjectFile(allow_interactive=True)` - interactive，连接到 GUI 会话

两者的会话 in-memory 仿真结果**不自动共享**。

#### 验证情况
| 测试 | 方式 | 结果 |
|------|------|------|
| 诊断脚本（源文件） | modeler+results 同路径，交互模式 | ✅ S11=[-0.88dB] |
| 副本脚本（副本文件） | 同上 | ✅ S11=[-1.03dB] |
| orchestrator 端到端 | 同上 | ❌ tree path not found |

#### 结论
CST COM 会话管理策略导致行为不一致。诊断/副本脚本在干净环境（无历史 GUI 残留）下可 work；orchestrator 由于 Streamlit/多次调用累积效应，更易触发会话混乱。

### Real Mode 根因确认 ✅

历史运行日志（`task_022`, `task_027`）揭示真实错误：
- 错误信息：`"Project is opened in CST Studio Suite. Consider using the setting 'allow_interactive=True'"`
- 根因：工程文件被 CST GUI 锁定，`allow_interactive=False` 无法关联到已打开的 session
- 这解释了为什么 modeler/results 分属不同 COM session 却都能访问文件 —— GUI 持有锁，非 interactive 模式无法打开

**修复方案**：
1. `CSTAdapter.open()`：results 层使用 `allow_interactive=True` 关联到 GUI session
2. `CSTAdapter.run_simulation()`：仿真后强制 `save_project()`，刷新 results session
3. `default_config.json`：`save_project_after_simulation: true`
